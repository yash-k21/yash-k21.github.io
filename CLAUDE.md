# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this is

Yash Khaitan's personal academic website (`yash-k21.github.io`), built on the [al-folio](https://github.com/alshedivat/al-folio) Jekyll theme. It's a static site: Jekyll (Ruby) generates HTML from Markdown/Liquid templates and YAML data files, deployed to GitHub Pages via GitHub Actions.

## Commands

**Local dev (Docker, recommended):**
```bash
docker compose up          # serves at http://localhost:8080, live-reloads on change
docker compose up --build  # rebuild image after Gemfile/package.json changes
```

**Local dev (native Ruby, no Docker):**
```bash
bundle install
bundle exec jekyll serve   # serves at http://localhost:4000
```

**Build (production):**
```bash
JEKYLL_ENV=production bundle exec jekyll build   # outputs to _site/
```

**Formatting (Prettier + Liquid plugin), CI-enforced on every push/PR:**
```bash
npx prettier . --check
npx prettier . --write
```

**Manual GitHub Pages deploy** (pushes prebuilt `_site` to `gh-pages` branch — normally unnecessary since CI auto-deploys on push to `main`):
```bash
bin/deploy
```

There is no JS/Ruby test suite. CI instead runs: Prettier formatting check, a broken-link checker (lychee) against the built site, CodeQL, and an optional manual Axe accessibility check (see `.github/workflows/`).

## Architecture

**Content lives in data/markdown, not code.** Almost all day-to-day edits (adding a paper, a news item, updating the CV) are data changes, not template changes:

- `_bibliography/papers.bib` — publications, consumed by `jekyll-scholar`. Config for how it's rendered (style, filters, badges) is in the `scholar:` block of `_config.yml`. Extra per-entry fields like `pdf`, `code`, `website`, `abstract`, `bibtex_show` are supported directly in the BibTeX entry.
- `_data/cv.yml` — CV content (fallback source; `assets/json/resume.json` takes precedence if present).
- `_data/coauthors.yml`, `_data/venues.yml` — lookup data cross-referenced from bibliography entries (e.g. to link co-author names or venue abbreviations).
- `_data/socials.yml` — social links and the Google Scholar user ID used by `bin/update_scholar_citations.py` and the citation-badge plugins.
- `_data/citations.yml` — cached Google Scholar citation counts, generated by `bin/update_scholar_citations.py` (not hand-edited).
- `_data/repositories.yml` — GitHub usernames/repos shown on the `/repositories/` page.
- `_news/` — collection of short announcements shown on the homepage.
- `_projects/` — collection shown on the `/projects/` grid.
- `_books/` — book-review collection (custom collection added on top of stock al-folio).
- `_pages/` — top-level site pages (`about.md`, `cv.md`, `publications.md`, `projects.md`, `books.md`, etc.); each sets its `layout` in front matter.
- `_config.yml` — central site configuration: identity fields, theme/layout toggles, `scholar:` (bibliography) settings, `collections:` definitions, social/analytics integrations.

**Rendering layers**, only touch these for structural/visual changes:

- `_layouts/` — page-level templates (`about.liquid`, `cv.liquid`, `bib.liquid` for individual publication entries, `post.liquid`, `distill.liquid` for distill.pub-style posts, `book-review.liquid`, `book-shelf.liquid`, etc.). Pages/collection items pick one via front-matter `layout:`.
- `_includes/` — reusable partials pulled into layouts (e.g. `header.liquid`, `footer.liquid`, `citation.liquid`, `selected_papers.liquid`, `repository/`, `cv/`, `resume/` subfolders).
- `_sass/` — SCSS partials; `_sass/_themes.scss` has the `--global-theme-color` variable and other theme colors, `_sass/_variables.scss` lists available color names.
- `_plugins/` — custom Ruby Jekyll plugins, notably `google-scholar-citations.rb` and `inspirehep-citations.rb` (render citation badges), `hide-custom-bibtex.rb`, `cache-bust.rb`, `details.rb` (publication "Details" panel), `external-posts.rb`, `remove-accents.rb`, `download-3rd-party.rb`.
- `_scripts/` — client-side JS setup snippets injected via config toggles (analytics, giscus comments, photoswipe, search).
- `assets/` — static files: `pdf/` for paper PDFs, `img/`, `json/resume.json`, `bibliography/` (per-paper "details" pages), `jupyter/`, `plotly/`, etc.

**Data flow example** (adding a publication): add/edit a BibTeX entry in `_bibliography/papers.bib` → jekyll-scholar renders it via the `bib.liquid` layout on `/publications/` → citation badges are populated by the `_plugins/*-citations.rb` plugins using `_data/socials.yml` (scholar user ID) and cached counts in `_data/citations.yml`.

## Notes

- Prettier formatting is enforced in CI for all files (via `@shopify/prettier-plugin-liquid`); run it before pushing to avoid CI failures. Excluded paths are listed in `.prettierignore`.
- `_data/citations.yml` is machine-generated (`bin/update_scholar_citations.py` scrapes Google Scholar) — don't hand-edit it, and note it's excluded from Prettier.
- Pushes to `main` that touch site-relevant paths auto-trigger the `Deploy site` GitHub Action, which builds with Jekyll, runs `purgecss`, and deploys `_site` to GitHub Pages. No manual deploy step is normally needed.
